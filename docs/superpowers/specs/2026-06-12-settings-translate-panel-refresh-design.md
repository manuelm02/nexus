# Nexus Sidebar / Settings / Translate Panel Refresh Design

**Date:** 2026-06-12  
**Status:** Approved for execution  
**Audience:** 实施工程师 / DeepSeek / 前端协作代理  
**Scope:** `Sidebar` 底部账户区、`Settings` 模型配置页、`Translate` 输入与历史记录区重构  

---

## 1. 背景与问题定义

当前前端已经形成了统一的 Navy Mono / Quiet Knowledge OS 基调，但这三个区域的体验仍然不稳定：

- `Sidebar` 左下角把 `Settings`、用户入口、`退出登录` 当成三个平级块堆在一起，信息层级混乱，视觉语言也不统一。
- `Settings` 仍停留在“单模型表单 + 次级配置”的早期形态，无法自然表达“多模型 + 唯一默认模型 + 工作流覆盖”的真实业务规则。
- `Translate` 已有可用业务能力，但输入区过大、历史工具条松散、历史卡片密度偏低，不像一个高频工具页面。

本次重构只做前端信息架构、交互与视觉层重构，不改变以下业务前提：

- 仍使用当前路由结构，不新增 `/m/*` 之类移动端路由。
- 仍保持 `TranslatePage` 作为共享状态编排层，桌面端和移动端共用业务逻辑。
- 仍复用已有 `LlmProvider`、`WorkflowLlmConfig`、翻译历史分页接口与 401 refresh 机制。

---

## 2. 设计目标

### 2.1 总体目标

让这三个区域都更像“长期使用的私人 AI 工具面板”，而不是临时拼出来的表单与卡片集合。

### 2.2 成功标准

- `Sidebar` 底部从三个分散入口变成一个统一的账户胶囊区。
- `Settings` 明确表达“全局默认模型”与“工作流覆盖”的关系。
- `Settings` 中每个 provider 都能在列表级别完成浏览和核心操作。
- `Translate` 原文输入框默认仅两行起步，但可随内容自动增高。
- `Translate` 历史工具条更像一条检索/分页控制带，而不是散放的几个表单项。
- `Translate` 历史卡片更紧凑，只保留高频扫读所需的信息。

---

## 3. 风格方向

### 3.1 保留的基调

继续沿用现有设计系统：

- 视觉气质：`Quiet Knowledge OS`
- 主色方向：深海军蓝 + 白色工作面 + 低干扰灰蓝中性色
- 交互档位：`L1`

### 3.2 本次局部增强

虽然总体基调不变，但允许这些页面更“面板化”：

- 更清晰的区块边界
- 更明确的状态徽章
- 更短、更紧的卡片节奏
- 更像工具栏的控制区

不允许引入：

- 花哨渐变标题
- 重动画或滚动特效
- 过强营销感卡片
- 过多装饰性色块

---

## 4. 信息架构

## 4.1 Sidebar

### 当前问题

- `Settings`、`Profile`、`退出登录` 三个模块彼此割裂。
- 用户看到的是三个独立按钮，而不是“当前账户/工作区”的一个统一模块。
- `Profile` 与 `Settings` 的边界不清晰，导致左下角显得像一个补丁区。

### 新结构

底部区域只保留一个主容器：`Account Capsule`

容器内部结构：

1. 账户身份区
2. 操作区

#### 账户身份区

展示内容：

- 圆形头像
- 昵称或用户名
- 小号身份说明，例如 `Personal workspace`

交互规则：

- 点击整个身份区进入 `Profile`
- hover 时整体高亮，而不是只高亮头像

#### 操作区

只保留两个动作：

- `Settings`
- `退出登录`

规则：

- `Settings` 是普通系统动作，视觉上使用 quiet utility button
- `退出登录` 是危险动作，但默认不使用大红底；只在 hover/focus 时加强 destructive affordance

### 明确取消

- 不再保留一个单独占位的 `Profile` 按钮卡片
- 不再出现 `Settings / Profile / Logout` 三段式堆叠

---

## 4.2 Settings

### 当前问题

- 顶部 `LLM 配置 / 系统参数` tab 太轻，无法支撑当前最重要的模型管理任务。
- 默认模型的唯一性规则藏在创建表单 checkbox 中，不够清晰。
- provider 列表只有展示和删除，没有形成完整的“管理面板”感。
- `Translate` 的模型覆盖配置存在，但还没有被表达成“工作流覆盖”概念。

### 新信息架构

页面按纵向连续面板组织，不再依赖顶部 tab 做主要导航。

推荐顺序：

1. 页面头部说明
2. `默认模型` 面板
3. `Provider 列表` 面板
4. `工作流覆盖` 面板
5. `系统参数 / Jobs / Tasks` 次级面板

---

## 5. Settings 详细规范

## 5.1 页面头部

保留：

- eyebrow：`System control`
- 标题：`Settings`

新增要求：

- 增加一段一句话说明，明确这是“模型路由与系统参数”的管理入口
- 头部右侧可放一个轻量状态说明 badge，例如 `1 default required`

目的：

- 第一眼就告诉用户这不是杂项页，而是“系统配置工作台”

## 5.2 默认模型面板

该面板回答一个核心问题：当其他模块没有指定模型时，系统用谁工作？

必须清楚表达：

- 默认模型是全局兜底模型
- 任何时刻都必须有且仅有一个默认模型
- 默认模型不是创建表单上的一个顺手 checkbox，而是一条明确的全局规则

建议内容：

- 当前默认 provider 名称
- provider 类型
- model 名称
- 启用状态
- 简短说明文案

推荐交互：

- 列表项内提供 `设为默认`
- 当前默认项显示明显但不过度夸张的 `默认` badge

## 5.3 Provider 列表面板

该面板是本页核心视觉区块。

每张 provider 卡片建议包含：

- 名称
- provider 类型
- model
- 启用 / 禁用状态
- 是否默认
- 关键操作：`编辑`、`设为默认`、`删除`

视觉要求：

- 卡片更像“条目面板”，不是大块展示卡
- 默认项应有轻强调背景或边框
- 禁用项降低对比度，但不能影响信息可读性

交互要求：

- `编辑` 优先级高于新增；用户应该能自然地修改已有项
- `设为默认` 只出现在非默认项上，或在默认项上显示 disabled state
- `删除` 必须保留确认语义

### 新增 / 编辑表单策略

推荐采用折叠式或抽屉式表单，而不是让大型表单始终占据页面主空间。

推荐行为：

- 点击 `添加 Provider` 展开内联编辑区或打开轻量 side panel
- 点击某一项的 `编辑` 时复用同一套表单状态

表单字段：

- `name`
- `provider`
- `apiKey`
- `model`
- `baseUrl`（仅当 provider 需要时）
- `enabled`

说明：

- `是否默认` 不再作为创建表单中的核心入口，而应由列表层操作主导

## 5.4 工作流覆盖面板

当前只需要服务 `Translate`，但命名和结构必须为未来扩展预留空间。

推荐标题：

- `工作流覆盖`
- 或 `Workflow Overrides`

当前只展示一条配置：

- `Translate`

字段表现：

- 左侧：工作流名称与一句说明
- 右侧：下拉框

下拉框选项：

- `继承全局默认`
- 某个具体 provider

推荐规则：

- `继承全局默认` 作为默认和推荐选项
- 如果选择了具体 provider，要让用户清楚这是覆盖行为，而不是全局变更

## 5.5 次级系统面板

`系统参数` 与 `Jobs / Tasks` 保留，但降级为页面后半区的次级区域。

原因：

- 当前用户进入 `Settings` 的最主要任务已经是模型管理
- `Jobs / Tasks` 和系统参数不应抢主视觉

表现方式：

- 视觉更轻
- 区块标题更小
- 与模型面板之间用清晰分隔拉开层级

---

## 6. Translate 页面详细规范

## 6.1 页面目标

Translate 页面应该像一个“高频翻译工作台”，而不是一张大输入表单。

工作流顺序保持：

1. 输入原文
2. 选择目标语言 / 风格
3. 执行翻译
4. 查看结果
5. 回填历史记录

但视觉上必须更紧凑、更快进入状态。

## 6.2 原文输入区

### 核心要求

- 默认仅显示两行高度
- 随输入内容自动增高
- 禁止用户手动拖拽 resize

### 设计原因

- 绝大多数翻译请求在起始阶段只需要一句话或一小段内容
- 大输入框会让页面显得笨重，也会压缩结果区与历史区
- 自动增高能兼顾长文本复制场景

### 桌面端建议

- 输入框上方保留标题与字符数
- 输入框本体像“工作条中的主编辑区”，不是传统大 textarea
- 控件与输入框之间的垂直间距缩短

### 移动端建议

- 顶部输入卡保持单列
- 文本区起始高度也为两行左右，但保留舒适触控 padding

## 6.3 控制条

控制条顺序固定为：

1. 目标语言
2. 风格
3. 翻译按钮

### 桌面端

- 同一行排布
- 目标语言 Select 为固定宽度
- 风格优先使用紧凑 chips / segmented controls
- 翻译按钮固定在右端

### 移动端

- 可拆成两行
- 但信息顺序不变
- 风格项可横向滚动

## 6.4 结果区

结果区不需要本次大改业务结构，但要保持与新输入区的紧凑节奏一致。

要求：

- 结果区紧跟输入区之后
- provider 异常提示集中在结果区处理
- 正常状态下不要出现多余的 `ready` 提示

目的：

- 让“异常时才提醒”成为统一原则

## 6.5 历史工具条

当前搜索框、每页条数与分页模块彼此独立，显得零碎。

重构目标：

- 把它们收进同一个紧凑工具条容器
- 形成“检索 + 翻页”的统一控制带

建议内容：

- 搜索框
- 每页条数 Select
- 分页状态
- 上一页 / 下一页

桌面端：

- 优先一行对齐

移动端：

- 可折成两行
- 但保持同一视觉容器

## 6.6 历史记录卡片

### 当前问题

- 卡片高度偏大
- 留白偏多
- 信息量没有高到需要那么大的面积

### 新卡片目标

做成可快速扫描的紧凑记录流。

每张卡只展示必要信息：

- 原文摘要
- 译文摘要
- 时间
- 目标语言
- 风格
- 删除动作

### 桌面端样式建议

- 两列或单列都可，但单卡高度必须明显缩小
- 删除按钮默认弱化，hover 时增强
- 点击整卡仍然回填

### 移动端样式建议

- 单列堆叠
- 删除按钮可直接可见
- 保持 44px 可触控目标

### 明确不展示

- 非关键长说明
- 冗余大标签
- 大块 provider 装饰

## 7. 响应式策略

本次严格遵守项目既有规范：同一路由、业务共享、视图按复杂度拆分。

### Sidebar

- 桌面端实现账户胶囊区
- 移动端不复制这套桌面结构，但应保证账户与退出逻辑不出现重复入口冲突

### Settings

- 桌面端可使用更明显的分区卡片
- 移动端必须单列堆叠，避免硬压成两栏
- 默认模型、Provider 列表、工作流覆盖的顺序在两端保持一致

### Translate

- 保持 `TranslatePage` 共享 query / mutation / state
- 只拆分视图和交互结构
- 桌面端强调横向工具条
- 移动端强调单手操作与纵向节奏

---

## 8. 不在本次范围内

以下内容本次不改：

- 后端接口结构
- `LlmProvider` / `WorkflowLlmConfig` 数据模型
- 认证、token refresh 流程
- `Translate` 的流式翻译业务逻辑
- 新增更多 workflow override 项
- 新建移动端专属路由

## 9. 实施注意事项

### 9.1 代码结构

- 业务逻辑继续放在 `frontend/src/pages/Settings/index.tsx` 和 `frontend/src/pages/Translate/index.tsx`
- 复杂视图应继续拆分成桌面/移动组件
- Provider 编辑、默认切换、删除确认等交互若开始变复杂，应抽成局部子组件

### 9.2 注释要求

必须遵守项目 AGENTS.md 中的注释规范：

- 每个导出的 React 组件顶部加一行中文注释说明用途
- 复杂 `useEffect` / `useQuery` / `useMutation` 需要解释触发条件和副作用
- 与默认模型唯一性、workflow 覆盖、textarea 自动增高相关的非显而易见逻辑必须写明 why

### 9.3 视觉一致性

- 不允许硬编码新业务色值
- 继续复用 `DESIGN.md` 中的 CSS variables、按钮和 surface 规则
- 不要把 `Settings` 做成强营销风格的 SaaS 页面

## 10. 验收清单

- `Sidebar` 左下角已重构为统一账户胶囊区
- `Profile` 不再作为底部独立第三块入口出现
- `Settings` 已能清楚区分默认模型、Provider 列表、工作流覆盖、系统参数
- Provider 列表支持编辑、删除、设为默认、状态展示
- `Translate` 输入框默认两行起步并自动增高
- `Translate` 历史工具条已整合为统一控制带
- `Translate` 历史卡片已压缩并只保留必要信息
- 桌面端和移动端都保持同一路由与共享业务逻辑
