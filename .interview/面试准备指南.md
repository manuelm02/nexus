## 第一部分：Nexus 项目亮点解析（面试素材库）

### 1.1 项目一句话定位

> "一个个人 AI 工作台，以六边形架构为骨架，集成多 LLM Provider 动态路由、RAG 知识库、异步任务引擎、多源 Web 爬虫，涵盖 GTD/财务/翻译/内容雷达等 AI 增强模块。"

这句话在面试自我介绍时要烂熟于心，它同时展示了：架构能力、AI 集成经验、业务落地能力。

---

### 1.2 六个可深挖的核心亮点

#### 亮点 ① — 六边形架构（Port/Adapter）应用于 AI 基础设施

**项目证据**
- `MindBankPort` / `WebCrawlerPort` 定义端口契约
- `AnythingLlmMindBank`、`JinaReaderCrawler`、`BiliBiliCrawler` 等是适配器实现
- `@ConditionalOnProperty` 实现运行时切换知识库后端

**面试话术**
> "我在项目里用了六边形架构来隔离 AI 基础设施。
> 比如知识库存储，定义了 `MindBankPort` 接口（ingest / search / delete），
> 当前实现是 AnythingLLM，但如果要切到 Milvus 或 pgvector，
> 只需要新增一个 Adapter，业务层零改动。
> 爬虫层同理，Jina Reader / 小红书 / B站 各自独立，
> 通过 `WebCrawlerDispatcher` 按 URL 匹配自动路由。"

**为什么这是亮点**：大部分写 AI 应用的开发者把 LLM 调用和业务逻辑耦合在一起，你这里有清晰的层次隔离，面试官会认为你有系统设计意识。

---

#### 亮点 ② — 多 LLM Provider 动态路由 + 工作流级模型绑定

**项目证据**：`LlmConfigService.resolveModel(workflowType)`

**核心设计**
```
调用方 → resolveModel("prism")
         ↓
         查 workflow_llm_configs（工作流绑定）
         ↓ 未绑定
         降级到全局默认 Provider
         ↓
         buildModel() → 根据 provider 类型动态构造 LangChain4j 实例
```

**面试话术**
> "系统支持 OpenAI / DeepSeek / Anthropic / Ollama 四种 Provider，
> 翻译模块用便宜的 DeepSeek，Forge 对话用 Claude，
> 通过数据库配置动态切换，不需要重启服务。
> API Key 用 AES-256 加密落库，前端接口加了 `@JsonIgnore`，
> 做到了密钥的全生命周期保护。"

**延伸问题准备**：面试官可能追问"为什么用 ECB 不用 GCM"——代码里有注释，ECB 对短字符串（API Key）安全性足够，如需更高安全可升级 GCM，说明你是有意识地做了权衡而非无知选择。

---

#### 亮点 ③ — 异步任务状态机（AI 长任务通用模式）

**项目证据**：`Task` 实体 + `TaskService` + `AsyncConfig`（`workflowExecutor` 线程池）

**状态流转**
```
pending → running → completed
                  → failed
```

**面试话术**
> "AI 任务普遍耗时 10~60 秒，不能用同步接口。
> 我设计了一个通用异步任务引擎：
> 前端 POST 创建任务拿到 taskId，然后轮询 GET /tasks/{id} 查状态。
> 后端用专属线程池（`workflowExecutor`，核心4/最大16）执行 LLM 调用，
> 结果写回 result_text / result_markdown 字段。
> 任务有过期时间，定时任务负责清理。"

**为什么这是亮点**：这是 AI 应用的标准工程模式，等同于"异步工作队列"的轻量版本，面试官会认可你的工程化思维。

---

#### 亮点 ④ — 多源 Web 爬虫 + 策略调度

**项目证据**：`WebCrawlerDispatcher` + `@Order` 优先级 + `supports(url)` 路由

**设计模式**：策略模式 + 责任链（按优先级匹配第一个能处理的爬虫）

**面试话术**
> "内容雷达模块需要抓取任意 URL，但不同平台需要不同策略：
> 小红书需要特殊解析，B站有视频页面，通用网页用 Jina Reader 转 Markdown。
> 我用策略模式+优先级调度，新增平台只需实现 `WebCrawlerPort` 接口即可，
> 完全符合开闭原则。"

---

#### 亮点 ⑤ — Spring WebFlux WebClient 非阻塞 HTTP

**项目证据**：`JinaReaderCrawler` 和 `TelegramNotificationService` 都用 `WebClient`，非 `RestTemplate`

**面试话术**
> "调用外部 API（Jina Reader、Telegram Bot）用了 WebFlux 的 WebClient，
> 相比 RestTemplate 是非阻塞的，在爬虫并发场景不会堵塞线程。
> 配合 `.timeout(Duration.ofSeconds(30))` 做了超时保护，防止外部服务拖垮系统。"

---

#### 亮点 ⑥ — 全栈能力展示（Java + React + 运维）

- 后端：Spring Boot 3 / MyBatis-Plus / Flyway / JWT / Spring Security
- 前端：React 18 / TanStack Query / Zustand / shadcn/ui
- 基础设施：Docker Compose / Caddy / PostgreSQL / MinIO
- AI 集成：LangChain4j / AnythingLLM / Jina Reader

**面试话术**：可以说自己独立完成了从数据库设计、后端 API、前端 UI 到 Docker 部署的全链路，这在小团队和 AI 创业公司是极有价值的能力。

## 第二部分：值得深入研究的技术方向（职业护城河）

以下方向是 2026 年 AI 应用开发岗位最被看重的，结合你的项目有天然的延伸路径。

### 2.1 RAG（检索增强生成）—— 最核心的AI应用能力

**为什么重要**：90%的企业 AI 应用都是 RAG，面试必问。

**Nexus 中的延伸机会**：`MindBankPort` 就是 RAG 的 ingestion + retrieval 层，把它做完整：
1. 文档分块（Chunking）策略：固定长度 / 语义分割
2. Embedding 向量化：调用 OpenAI `text-embedding-3-small` 或本地 Ollama
3. 向量数据库：直接用 PostgreSQL + pgvector 扩展（不引入新组件）
4. 相似度检索 + Reranking
5. 上下文窗口组装 → 调用 LLM 生成答案

**面试价值**：能完整描述 RAG pipeline 的候选人在市场上仍属少数。

---

### 2.2 Function Calling / Tool Use —— Agent 的核心机制

**为什么重要**：Agent = LLM + 工具调用能力，工具调用是 Agent 和普通 chatbot 的本质区别。

**Nexus 中的延伸机会**：`ForgeService`（Forge 模块）可以做成一个有工具的 Agent：
- 工具1：搜索 Mindbank（调 RAG）
- 工具2：创建 Focus 任务
- 工具3：查询 Ledger 订阅信息
- 工具4：抓取 URL 内容

LangChain4j 的 `@Tool` 注解 + `AiServices` 接口就能实现。

---

### 2.3 Streaming 流式响应 —— 用户体验关键

**为什么重要**：LLM 生成慢，不流式输出用户体验很差，实际产品几乎都要做。

**技术实现**：Spring Boot 的 SSE（Server-Sent Events）+ LangChain4j 的 `StreamingChatLanguageModel`

**面试价值**：能说清楚 SSE vs WebSocket 的取舍，能解释 `@Async` + `SseEmitter` 的配合方式，这是工程细节，加分项。

---

### 2.4 Prompt Engineering —— 可立即提升的软实力

**核心概念**（需要能说清楚）：
- System / User / Assistant 角色分工
- Few-shot 示例的作用
- Chain of Thought（思维链）
- Temperature 和 Top-P 的调节原则
- 结构化输出（JSON mode）

**Nexus 中的实例**：`PrismService.buildPrompt()` 就是 Prompt Engineering，可以深化：多加 few-shot 示例、加 CoT 指令、测试不同风格输出。

---

### 2.5 向量数据库基础概念

**面试必知**：pgvector（已在你的 PostgreSQL 中可用）、Milvus、Chroma、Pinecone。
- 余弦相似度 vs 欧式距离
- 近似最近邻（ANN）算法：HNSW、IVF
- 为什么向量搜索比 MySQL LIKE 查询更适合语义场景

### 3.2 AI / LLM 核心概念（从入门到够用）

#### LLM 基础

| 概念 | 需要掌握的程度 |
|------|-------------|
| Token 是什么 | 能解释为什么"长文本更贵"，为什么要做文本压缩 |
| Temperature / Top-P | 0→确定性输出，1→多样性输出，翻译用低值，创意写作用高值 |
| Context Window | 为什么 RAG 要控制 chunk 大小 |
| System Prompt vs User Prompt | 角色设定在 system，用户输入在 user |
| 幻觉（Hallucination） | 产生原因 + RAG 如何缓解 |

---

#### RAG 完整 Pipeline（重中之重）

```
原始文档
   ↓  文档加载（PDF/网页/文本）
   ↓  分块（Chunking）：固定长度 / 语义切分
   ↓  Embedding（向量化）：text-embedding-3-small 等
   ↓  存入向量数据库（pgvector / Milvus）
          ↓
用户提问
   ↓  Query Embedding
   ↓  向量相似度检索（Top-K）
   ↓  可选：Reranking（Cross-encoder 重排序）
   ↓  组装 Prompt（上下文 + 问题）
   ↓  调用 LLM 生成答案
```

**面试话术准备**：
> "RAG 的核心瓶颈有两个：检索质量和上下文组装。
> 检索质量靠 chunk 策略和 embedding 模型选择；
> 我们项目用 AnythingLLM 作为知识库后端，通过 MindBankPort 接口隔离，
> 后续可以平滑切换到自建的 pgvector 方案。"

---

#### Agent 基础

| 概念 | 说明 |
|------|------|
| ReAct 模式 | Reason + Act 交替：思考 → 选工具 → 观察结果 → 再思考 |
| Function Calling / Tool Use | LLM 根据对话决定调用哪个函数，参数由 LLM 填充 |
| LangChain4j `@Tool` | Java 方法加注解就能成为 Agent 可用的工具 |

### 3.3 系统设计题（面试高频，需要准备）

#### 题型1：设计一个 AI 文档问答系统

**参考回答框架**：
1. **文档处理管线**：上传 → 解析 → 分块 → Embedding → 存向量库
2. **查询链路**：用户问题 → Embedding → 向量检索 → 重排序 → 组装 Prompt → LLM → 流式输出
3. **存储层**：关系数据库（文档元数据）+ 向量数据库（chunks）+ 对象存储（原始文件）
4. **并发处理**：大文档解析放异步任务队列（你的 TaskService 就是）
5. **成本控制**：缓存高频查询的 Embedding，复用 system prompt

#### 题型2：设计一个多 LLM Provider 管理系统

**你直接可以说 Nexus 的实现**：Provider 配置 + 工作流绑定 + 运行时解析 + API Key 加密。

#### 题型3：LLM 调用超时/失败如何处理

1. 重试机制（exponential backoff）
2. 降级策略（主用 Claude，降级用 GPT-4o mini）
3. 超时设置（避免线程长时间阻塞）
4. 异步任务 + 状态机（你的 Task 就是）

## 附录：高频面试问题速查

### Spring 必背

1. **`@Transactional` 失效场景**
   - 同类内部调用（未走代理）
   - 方法非 public
   - 异常被 catch 吞掉
   - 非受检异常（默认只回滚 RuntimeException）
   - 数据库引擎不支持事务（如 MyISAM）

2. **Bean 的生命周期**（9步）
   实例化 → 属性注入 → BeanNameAware → BeanFactoryAware → ApplicationContextAware → BeanPostProcessor前置 → InitializingBean/init-method → BeanPostProcessor后置 → 使用 → DisposableBean/destroy-method

3. **`@Autowired` 注入顺序**
   byType → 有多个时 byName → `@Qualifier` 指定

### 数据库必背

1. **索引失效场景**：`%` 前缀 LIKE / 对索引列做函数 / 类型隐式转换 / OR 连接非索引列 / `!=` `<>` / IS NULL

2. **事务隔离级别**：
   - Read Uncommitted → 脏读
   - Read Committed → 不可重复读（Oracle默认）
   - Repeatable Read → 幻读（MySQL默认）
   - Serializable → 性能最差

3. **MVCC 实现**：每行数据有 `trx_id` 和 `roll_pointer`，Read View 决定哪个版本可见

### LLM 必背

1. **RAG vs Fine-tuning 怎么选**：
   - RAG：知识频繁更新、需要引用来源、企业私有知识库 → 选 RAG
   - Fine-tuning：固定任务风格/格式、需要模型行为改变 → 选 Fine-tuning

2. **Chunk 大小如何选**：
   - 太小：语义不完整，检索不准
   - 太大：噪声多，超出 context window
   - 实践：512~1024 tokens，加 128 tokens 重叠（overlap）

3. **向量搜索慢怎么优化**：
   - 减少向量维度（用轻量 embedding 模型）
   - 使用 ANN 索引（HNSW）而非暴力搜索
   - 分域（domain）隔离，缩小搜索范围（你的 `MindBankPort` 有 domain 字段）
